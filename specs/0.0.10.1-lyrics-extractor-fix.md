# MCC 0.0.10.1 — LyricsExtractor Fix (Single-Hypothesis + Streaming Append)

Patch release. Fixes two related defects in `LyricsExtractor.transcribe(...)` surfaced by Sanctuary device tests in early-to-mid May 2026. Public API surface is unchanged; behavior changes are bounded to the SFSpeechRecognizer integration path and validated by new tests.

Tier 2 release (patch on existing 0.0.10 subsystem; no new public types, no breaking changes). Tagged `0.0.10.1`.

---

## Problem statement

### Defect 1 — multi-hypothesis flattening produces duplicated transcripts

Sanctuary device test on **2026-05-12** sang a ~32-second song fragment ("Wakin up with you in the mornin / Feels like a warm summer / Comin home to you in the evening / All my troubles melt away / Whoa whoa oh oh oh oh oh / All my troubles melt away") and received a transcript that contained the song text repeated ~3 times with slight variation between iterations. The three repetitions ended differently:

- "...all my travels met my troubles me"
- "...how much troubles met my troubles me"
- "...all my troubles…"

This pattern — same text, different tails — is the diagnostic signature of `SFSpeechRecognizer` alternative hypotheses being flattened together into a single token stream.

Root cause: [`Sources/MusicCraftCore/Voice/LyricsExtractor.swift`](../Sources/MusicCraftCore/Voice/LyricsExtractor.swift) lines 74–83:

```swift
let tokens = result.transcriptions.flatMap { transcription in
    transcription.segments.map { segment in
        TranscribedToken(
            text: segment.substring,
            onsetTime: segment.timestamp,
            duration: segment.duration,
            confidence: nil
        )
    }
}
```

`SFSpeechRecognitionResult.transcriptions` is an ordered list of alternative recognition hypotheses, best-first. The current code includes **every** hypothesis. For a clip with three alternative interpretations, every word in the audio is emitted three times (with overlapping `onsetTime` values across the three groups).

### Defect 2 — long-buffer truncation

An earlier device test (a 56-second song) transcribed only the first ~25 seconds of audio. This is consistent with the documented behavior of `SFSpeechAudioBufferRecognitionRequest` when a single large buffer is submitted via one `append(...)` call followed immediately by `endAudio()`: the recognizer can produce a final hypothesis from the front of the buffer before processing the tail, particularly on longer clips. Apple's recommended pattern for non-real-time recognition of pre-recorded audio is to append the buffer in small chunks (each one a separate `AVAudioPCMBuffer`) and only call `endAudio()` after all chunks have been appended.

---

## Root causes

1. **`result.transcriptions.flatMap { ... }`** includes every alternative hypothesis. `result.transcriptions` is ordered best-first; we want only `result.transcriptions.first?.segments`.
2. **One-shot `append + endAudio`** on a single full-buffer `AVAudioPCMBuffer` is an unreliable pattern for clips longer than ~30–40 seconds. The Speech framework expects the audio source to feed buffers progressively, even when the source is a pre-recorded file. Chunked appends keep the recognizer's internal stream engaged across the full duration.

---

## Fix

### 1. Use only the first transcription

Replace the `flatMap` with a single-hypothesis map:

```swift
let tokens = (result.transcriptions.first?.segments ?? []).map { segment in
    TranscribedToken(
        text: segment.substring,
        onsetTime: segment.timestamp,
        duration: segment.duration,
        confidence: nil
    )
}
```

### 2. Chunked streaming append

Slice the input float buffer into 1-second chunks. For each chunk, build a separate `AVAudioPCMBuffer` and call `request.append(chunkBuffer)`. After all chunks are appended, call `request.endAudio()`. No pre-padding or post-padding — chunks slice the input exactly. The chunk size is a single constant:

```swift
private static let chunkSeconds: Double = 1.0
```

A 1-second chunk size is small enough to keep the recognizer streaming continuously and large enough to avoid per-chunk allocator overhead becoming dominant on long clips. The exact value is not load-bearing; values between 0.5s and 2.0s are equivalent in expected behavior.

## Test requirements

Two new tests in `Tests/MusicCraftCoreTests/AudioAnalysis/LyricsExtractorRealAudioTests.swift` (existing on-device test suite — already gated with `#if os(macOS)` skip, microphone permission check, and `SFSpeechRecognizer.isAvailable` guard, so the new tests inherit the same skip behavior on headless / unsupported hosts):

1. **`testSingleHypothesisShape`** — load a TTS fixture of moderate duration (the existing `long-passage` fixture, ~10s of synthesized speech). Transcribe via `LyricsExtractor.transcribe(...)`. Assert that the returned tokens have monotonically non-decreasing `onsetTime` values (multi-hypothesis flattening produces overlapping/repeating onset times across the hypothesis groups; single-hypothesis output is monotonic). Specifically: for each adjacent token pair `(t[i], t[i+1])`, `t[i+1].onsetTime >= t[i].onsetTime`. Tolerance: 0 (exact monotonicity). This test fails with the pre-fix `flatMap` implementation.

2. **`testFullDurationCoverage`** — concatenate the existing TTS fixtures (or use `long-passage`) to construct a buffer ≥30s. Transcribe. Assert that `tokens.last?.onsetTime + tokens.last?.duration` is within 2.0 seconds of the buffer's total duration. This validates that chunked streaming exercises the full clip; the pre-fix one-shot append behavior on long buffers truncates and fails this check.

If no concatenated long fixture is feasible, the test uses the longest available TTS fixture and asserts coverage of that duration (still meaningful for the chunk-streaming codepath, just shorter).

The existing `testLyricsExtractorAccuracy` aggregate test continues to pass without modification — the single-hypothesis fix should *improve* word-accuracy metrics (it removes the noise from alternative hypotheses being included), not regress them.

## Release rationale

Single-hypothesis return is the **correct semantic** for the current consumer surface. Sanctuary stores `audioAnalysis.lyricDraft` as a single string — there is no caller today that wants the alternatives list. If a future caller needs alternatives (e.g., a "did you mean…?" UI), MCC can add a separate entry point:

```swift
public static func transcribeWithAlternatives(
    buffer: [Float],
    sampleRate: Double,
    locale: String? = nil,
    configuration: Configuration? = nil
) async throws -> [[TranscribedToken]]
```

…returning one token-list per hypothesis. That's deferred; the `flatMap` behavior was a bug, not a feature.

Streaming append is the **idiomatic pattern** for pre-recorded audio. The fix aligns MCC with Apple's Speech framework guidance. No public API change is required to land the fix.
