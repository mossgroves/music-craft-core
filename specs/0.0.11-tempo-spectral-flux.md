# MCC 0.0.11 — TempoEstimator: Spectral Flux Onset Detection

Minor release. Replaces the internals of `TempoEstimator` (and its supporting `BeatTracker.detectBeats` buffer path) with a spectral-flux-based onset detector and histogram-based tempo aggregation. Public API is unchanged; algorithm correctness is validated against synthetic regression fixtures and existing real-audio fixtures.

Tier 2 release (algorithm change, no new public types, no API breakage). Tagged `0.0.11`.

---

## Problem statement

Phase 3.2 measurement (2026-04-29, recorded in [mcc.md](../../mossgroves-claude-workspace/mcc.md) and the cross-project log) showed `TempoEstimator` producing **0% accuracy on real audio fixtures** with a **systematic 1/3-tempo error**. Two pieces of evidence anchor this:

1. The Phase 3.2 GuitarSet measurement on 5 fixtures returned tempo estimates that were uniformly ~1/3 of the JAMS-annotated ground truth.
2. The Phase 3.3 diagnostic test ([`Tests/MusicCraftCoreTests/AudioAnalysis/diagnostics/Phase33TempoDiagnosticTests.swift`](../Tests/MusicCraftCoreTests/AudioAnalysis/diagnostics/Phase33TempoDiagnosticTests.swift)) reproduced the failure with a clear single-fixture trace (this session's run shows `Detected tempo: nil` from the buffer path even when chord detection and beat detection produce output).

Root cause is in the **buffer-derived path** (`TempoEstimator.estimateTempo(buffer:sampleRate:)` → `BeatTracker.detectBeats(buffer:sampleRate:)`):

- [`BeatTracker.swift:92-110`](../Sources/MusicCraftCore/DSP/BeatTracker.swift) `computeOnsetStrengthSignal` calls `vDSP_rmsqv` on each windowed frame. This emits **RMS energy per frame**, not onset strength. RMS energy rises and falls with every sustained note, every chord-internal articulation (strum, finger-pick), and every breath — it captures attack transients of intra-strum events, not beat-grid events.
- The downstream autocorrelation locks onto the dominant periodicity in this signal, which on percussive guitar is the inter-string articulation rate (often ~3x the beat rate for fingerpicked or arpeggiated patterns). This produces the observed 1/3-tempo systematic error.

The JAMS-fed path (`estimateTempo(beats:)`) is unaffected — it accepts pre-detected beat times directly and only computes the inter-beat-interval mean. The Phase 3.3 measurement showing 0% on buffer-derived but plausible results on JAMS-fed confirms the locus is in onset detection.

## Algorithm — spectral flux onset detection

### 1. STFT

Hann-windowed STFT, 1024 samples per frame, 512-sample hop. Output: 2D magnitude array `mag[frame][bin]` where each frame has 512 magnitudes (halfN of windowSize).

Implementation note: MCC already has `DSPUtilities.hannWindow(length:)` and the `vDSP_create_fftsetup` / `vDSP_fft_zrip` pattern in `ChromaExtractor`. We reuse the same FFT setup pattern as an internal detail of the new spectral-flux helper.

### 2. Spectral flux (Dixon 2006, half-wave rectified)

For each frame `m > 0`:

```
sf[m] = Σ_k max(0, mag[m][k] - mag[m-1][k])
```

This is the **positive-only** spectral flux: count only energy *increases* per bin, sum across bins. Onsets are events where multiple bins simultaneously gain energy; sustains and decays contribute nothing. This is the canonical formulation per Dixon's 2006 onset-detection survey.

### 3. Normalize

Divide `sf` by its max value so the signal falls in `[0, 1]`. If `max(sf) == 0` (silence), return no onsets.

### 4. Adaptive threshold

For each frame `m`, compute a local **median** of `sf` over a window of ±0.1 seconds (`±W` frames where `W = round(0.1 * sampleRate / hopSize)`). Threshold:

```
threshold[m] = median(sf[m-W ... m+W]) + δ
```

where `δ = 0.07` (tunable). Median is robust to occasional spikes; mean would let one big onset suppress the next.

### 5. Peak picking

A frame `m` is an onset if all three hold:

- `sf[m] > threshold[m]`
- `sf[m] > sf[m-1]` and `sf[m] > sf[m+1]` (local max)
- `sf[m]` is the maximum within ±50 ms (minimum onset gap — prevents double-counting on rounded peaks)

### 6. Onset times

```
t[i] = onsetFrameIndex[i] * hopSize / sampleRate
```

### 7. Tempo histogram

For each consecutive onset pair `(t[i], t[i+1])`:

- `ioi = t[i+1] - t[i]`
- `bpmCandidate = 60 / ioi`
- Also generate octave candidates: `2 * bpmCandidate` and `0.5 * bpmCandidate`

Bin all candidates into 1-BPM bins from 40 to 200 BPM. Each candidate contributes a weight of `1.0` to its bin. (We considered IOI-deviation weighting; testing showed equal weighting is simpler and equally effective on the regression fixtures.)

### 8. Smoothing

Smooth the histogram with a 3-bin moving average. Prevents single-frame spikes from outranking broader peaks.

### 9. Peak detection

Find the top peak in `[40, 200]` BPM range. Confidence:

```
confidence = peakValue / totalHistogramSum
```

Return up to `configuration.maxCandidates` (default 3) entries from the top peaks. The first candidate is the primary; subsequent candidates are `isHarmonic = true` if they are 0.5x or 2x the primary, `false` otherwise.

If the histogram is empty (zero or one onset detected), return an empty array.

## API

`TempoEstimator.estimateTempo(buffer:sampleRate:configuration:)` and `TempoEstimator.estimateTempo(beats:configuration:)` retain their existing signatures. The `Configuration` struct stays as-is — `onsetWindowSize`, `onsetHopSize`, `minTempoMs`, `maxTempoMs`, `maxCandidates`, `harmonicRatios` are all preserved.

`Configuration.onsetWindowSize` and `onsetHopSize` now configure the **STFT window** (default `1024 / 512`, lowered from the existing default `2048 / 1024` to match the algorithm's per-frame onset granularity). The Configuration default change is observable to consumers who pass `.default` explicitly via stored values; for callers using `Configuration()` no change is required. **This is the one user-observable Configuration default shift in 0.0.11.**

`Configuration.minTempoMs` / `maxTempoMs` translate to BPM bounds: `60000 / maxTempoMs ≤ bpm ≤ 60000 / minTempoMs`. With defaults `300ms / 3000ms`, this is `20 ≤ bpm ≤ 200`. We clamp the histogram range to `[40, 200]` BPM for the primary search to avoid spurious low-BPM peaks; the Configuration bounds widen this only when explicitly set.

`Configuration.harmonicRatios` is **retained but deprecated for the buffer path** — the spectral-flux algorithm generates octave candidates internally (2x, 0.5x) during histogram construction. The field is still respected by the `estimateTempo(beats:)` JAMS-fed path for backward compatibility. New code should rely on the algorithm's internal octave handling.

`TempoEstimate.confidence` now means "fraction of histogram evidence at this BPM." Doc comment is updated:

> Confidence in `[0, 1]`. Represents the fraction of inter-onset-interval evidence supporting this BPM. Consumers displaying tempo to end users should gate on `confidence >= 0.3` to suppress unreliable estimates on low-rhythm material (e.g., monophonic vocals without clear pulse).

`isHarmonic` semantics unchanged.

## Internal helpers

The new algorithm introduces two `internal` (module-scoped) helpers in the DSP subsystem. Neither is part of the public API:

- `SpectralFluxOnsetDetector` (file `Sources/MusicCraftCore/DSP/SpectralFluxOnsetDetector.swift`): pure function `static func detectOnsets(buffer:sampleRate:windowSize:hopSize:thresholdDelta:) -> [TimeInterval]`. Computes STFT, spectral flux, adaptive threshold, peak picking, and returns onset times.
- `TempoHistogram` (file `Sources/MusicCraftCore/DSP/TempoHistogram.swift`): pure function `static func estimate(onsets:minBpm:maxBpm:smoothingWindow:) -> [(bpm: Double, confidence: Double)]`. Builds the histogram, smooths, and returns ranked peaks.

`BeatTracker.detectBeats(buffer:sampleRate:)` is rewired to call `SpectralFluxOnsetDetector` for the onset signal but keeps its existing public API. Its private `computeOnsetStrengthSignal` is removed.

These are internal additions and rewires, classified yellow under MCC's autonomy rubric.

## Accuracy targets and validation

### Synthetic regression (the 1/3-bug fixture)

A new test fixture generates a synthetic click track at 120 BPM (one click per 0.5s, click = 50ms of 1kHz sine, 10s total). The pre-0.0.11 algorithm returns 40 BPM on this fixture (systematic 1/3 error). The post-0.0.11 algorithm must return primary BPM within ±5% of 120 (i.e., 114–126 BPM).

This is a hard regression gate. If this fixture fails, the algorithm is wrong.

### GuitarSet subset

The existing `GuitarSetTempoTests.testGuitarSetTempoAccuracy` runs `TempoEstimator.estimateTempo(beats:)` with JAMS-annotated beat times — this path is **unaffected by the algorithm change** (it consumes pre-detected beats). It should continue to pass at its existing thresholds (within ±5%: 55%, within ±10%: 75%).

A new test, `GuitarSetTempoBufferTests.testBufferDerivedTempoAccuracy`, runs `TempoEstimator.estimateTempo(buffer:sampleRate:)` on a subset of GuitarSet fixtures (the first 5 alphabetically: ensures determinism without requiring the full corpus to be downloaded). Target: ≥ 40% within ±10% of ground truth. This is an honest target — the prior baseline is 0%, and any positive movement is improvement; 40% on a small subset represents meaningful improvement without overclaiming.

If the actual measured accuracy on the 5-fixture subset is lower than 40%, document the measurement in the spec and the release commit and proceed. We're in experimentation mode; tuning iterates after measurement.

### Voice fixtures

Voice without clear rhythm should produce **low-confidence** estimates. A new test asserts that `TempoEstimator.estimateTempo(buffer:sampleRate:)` on a TTS-generated speech fixture (existing `long-passage` from `LyricFixture.all()`) returns either an empty array or a primary estimate with `confidence < 0.5`. This validates that the confidence gate is meaningful — Sanctuary will use the `≥ 0.3` threshold to suppress tempo display on monophonic vocal captures.

## Confidence semantics for consumers

`TempoEstimate.confidence` semantics for callers:

| Range | Interpretation | Recommended display behavior |
|---|---|---|
| `>= 0.5` | Strong rhythmic support | Show tempo. |
| `0.3–0.5` | Moderate support | Show tempo (default Sanctuary threshold). |
| `< 0.3` | Weak support | Hide tempo display. The audio likely lacks clear rhythm (vocal hum, sustained pad, etc.). |

Sanctuary's `MCCAnalyzer` will adopt `0.3` as the display threshold and write `audioAnalysis.tempo = nil` when below.

## Migration notes for consumers

### Cantus (Sanctuary)

`MCCAnalyzer` currently writes `tempoEstimates.first?.bpm` unconditionally. Post-0.0.11:

```swift
let tempoEstimate = tempoEstimates.first
if let estimate = tempoEstimate, estimate.confidence >= 0.3 {
    audioAnalysis.tempo = estimate.bpm
} else {
    audioAnalysis.tempo = nil
}
```

### Other consumers

If any consumer relied on the old `Configuration.harmonicRatios` field driving the buffer path's harmonic candidate set, they will see different output. The new algorithm generates 2x and 0.5x candidates internally; manual harmonic-ratio expansion at the call site is no longer needed.
