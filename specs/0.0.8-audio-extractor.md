# MCC 0.0.8 — AudioExtractor Design Specification

Design document for the next MusicCraftCore release. Drafted for Chris's review and Sanctuary's peer review per the diagnosis-plan-execute pattern.

Status: Chris approved 2026-04-25. Sanctuary review complete 2026-04-25 — greenlight on implementation, all five API confirmations clean, one slice-mapping correction incorporated below. Ready for phased implementation prompt generation.

---

## Purpose

0.0.8 establishes MCC's **offline analysis pipeline** — the architectural counterpart to the real-time pipeline already shipped in 0.0.5–0.0.7. Where real-time analysis processes one frame at a time for live UI, offline analysis processes a complete audio buffer with multi-frame averaging, segmentation, and refined detection.

The release ports Cantus's proven `AudioExtractor.swift` algorithm into MCC, plus extracts the onset detection and noise-calibration primitives into the DSP layer where they belong. It also adds `MelodyKeyInference` — a pitch-class-template-based key inference algorithm distinct from `ProgressionAnalyzer.inferKey` — because the offline pipeline naturally produces pitch class sequences, and Sanctuary's contour-based key extraction needs this exact algorithm.

The release also formalizes the **contour API** frozen in the 2026-04-23 cross-project log decision: `ContourNote` with `pitchSemitoneStep`, `parsonsCode`, `onsetTime`, `duration`, exposed as `AudioExtractor.Result.contour: [ContourNote]`.

This is the largest extraction since 0.0.6. Multiple novel concerns combine: a new subsystem (AnalysisPipeline gets populated for the first time), new DSP primitives (onset detection, calibration helpers), a new inference type (MelodyKeyInference), and a new public output type (ContourNote). Diagnosis-plan-execute applies; the implementation will be phased with explicit checkpoints between novel concerns.

## Scope

In scope: a public `AudioExtractor` type in the new `AnalysisPipeline` subsystem, a public `OnsetDetector` and `NoiseCalibrator` in the DSP subsystem, a public `MelodyKeyInference` in the MusicTheory subsystem, a public `ContourNote` value type, a public `DetectedNote` value type, a public `AudioExtractor.Result` bundling all extracted descriptors, plus tests covering all of the above.

Out of scope (deferred or consumer-side): everything in the "Out of scope" section below.

## Architectural placement

The 0.0.8 release populates the `AnalysisPipeline` subsystem for the first time. This is intentional — `AudioExtractor` is the canonical example of a higher-order pipeline that composes lower-level primitives (DSP, ChordDetection, MusicTheory) into a single offline-analysis surface.

```
┌───────────────────────────────────────────────────────────────┐
│ AnalysisPipeline (NEW in 0.0.8)                               │
│   AudioExtractor                                              │
│     ├─ takes [Float] PCM buffer + sample rate                 │
│     └─ returns AudioExtractor.Result                          │
│        ├─ chordSegments: [ChordSegment]                       │
│        ├─ key: MusicalKey?                                    │
│        ├─ contour: [ContourNote]                              │
│        ├─ detectedNotes: [DetectedNote]                       │
│        └─ duration: TimeInterval                              │
└───────────────────────────────────────────────────────────────┘
              │ composes
              ▼
┌───────────────────────────────────────────────────────────────┐
│ DSP                                                            │
│   PitchDetector (shipped)                                     │
│   ChromaExtractor (shipped)                                   │
│   FFT, windows, DSPUtilities (shipped)                        │
│   OnsetDetector (NEW in 0.0.8)                                │
│   NoiseCalibrator (NEW in 0.0.8)                              │
│                                                                │
│ ChordDetection                                                 │
│   ChordDetector, IntervalDetector (shipped)                   │
│                                                                │
│ MusicTheory                                                    │
│   ProgressionAnalyzer.inferKey (shipped — chord-based)        │

│   ContourNote, DetectedNote (NEW in 0.0.8)                    │
└───────────────────────────────────────────────────────────────┘
```

Consumer apps (Cantus, Sanctuary) take an audio buffer they've captured or read from a file via their own AVAudioFile / AVAudioEngine code, hand it to `AudioExtractor.extract(buffer:sampleRate:)`, and receive a `Result` they can render or pass to a Foundation Model.

## Public API surface

The full new public surface is described below. All types are namespaced under `MusicCraftCore` and follow the established conventions: explicit public initializers, Equatable + Hashable + Sendable conformance, no opaque results.

### AudioExtractor

The top-level entry point for offline analysis. Stateless namespace, mirrors `Transposer` and `ProgressionAnalyzer` convention.

```swift
public enum AudioExtractor {

    /// Extract chord segments, key, melodic contour, and detected notes from an audio buffer.
    /// Pure function. No I/O, no async, no AVFoundation.
    /// 
    /// - Parameters:
    ///   - buffer: Mono Float32 PCM samples.
    ///   - sampleRate: Sample rate in Hz (typically 44100 or 48000).
    ///   - configuration: Optional tuning. Defaults are calibrated for Cantus's nylon-string and Sanctuary's vocal/instrumental capture.
    /// - Returns: Bundled extraction result.
    public static func extract(
        buffer: [Float],
        sampleRate: Double,
        configuration: Configuration = .default
    ) -> Result

    public struct Configuration: Equatable, Hashable, Sendable {
        public let onsetMinGapMs: Double           // default 500
        public let onsetEnergyMultiplier: Float    // default 2.0 (vs running average)
        public let onsetEnergyFloor: Float         // default 0.005 (absolute floor)
        public let chromaWindowSize: Int           // default 8192
        public let chromaHopSize: Int              // default 4096
        public let earlyFrameAttackSkip: Int       // default 2
        public let earlyFrameWindowSize: Int       // default 8
        public let extractionMinConfidence: Double // default 0.25 with ≥5 frames, 0.35 otherwise
        public let silenceThreshold: Float         // default 0.001 (-60dB) for noise calibration
        
        public init(
            onsetMinGapMs: Double = 500,
            onsetEnergyMultiplier: Float = 2.0,
            onsetEnergyFloor: Float = 0.005,
            chromaWindowSize: Int = 8192,
            chromaHopSize: Int = 4096,
            earlyFrameAttackSkip: Int = 2,
            earlyFrameWindowSize: Int = 8,
            extractionMinConfidence: Double = 0.25,
            silenceThreshold: Float = 0.001
        )
        
        public static let `default`: Configuration
    }

    public struct Result: Equatable, Hashable, Sendable {
        /// Detected chord segments in playback order.
        public let chordSegments: [ChordSegment]
        /// Inferred musical key from the chord progression.
        public let key: MusicalKey?
        /// Melodic contour as a sequence of pitched note events with absolute timing.
        public let contour: [ContourNote]
        /// Individual detected notes (pre-contour, raw monophonic events).
        public let detectedNotes: [DetectedNote]
        /// Total duration of the analyzed buffer in seconds.
        public let duration: TimeInterval

        public init(
            chordSegments: [ChordSegment],
            key: MusicalKey?,
            contour: [ContourNote],
            detectedNotes: [DetectedNote],
            duration: TimeInterval
        )
    }

    public struct ChordSegment: Equatable, Hashable, Sendable, Identifiable {
        public let id: UUID
        public let startTime: TimeInterval
        public let endTime: TimeInterval
        public let chord: Chord
        public let confidence: Double
        public let detectionMethod: DetectionMethod

        public init(
            id: UUID = UUID(),
            startTime: TimeInterval,
            endTime: TimeInterval,
            chord: Chord,
            confidence: Double,
            detectionMethod: DetectionMethod
        )

        public enum DetectionMethod: String, Equatable, Hashable, Sendable, CaseIterable {
            case classifier
            case interval
            case agreement
        }
    }
}
```

The configuration is exposed because Sanctuary's vocal capture may want different onset thresholds than Cantus's nylon-string strums, and exposing the parameters now (with sensible defaults) is non-breaking and lets consumers tune without forking.

### ContourNote (frozen API, per 2026-04-23 log decision)

The melodic contour primitive. Combines symbolic relative pitch (Parsons code, signed semitone step) with absolute timing.

```swift
public struct ContourNote: Equatable, Hashable, Sendable {
    /// Signed semitone step from the previous note. Zero for the first note (no predecessor).
    public let pitchSemitoneStep: Int
    /// Symbolic direction — Up, Down, Repeat. Equivalent to sign(pitchSemitoneStep) for non-first notes.
    public let parsonsCode: ParsonsCode
    /// Onset time in seconds from the start of the analyzed buffer.
    public let onsetTime: TimeInterval
    /// Note duration in seconds.
    public let duration: TimeInterval

    public init(
        pitchSemitoneStep: Int,
        parsonsCode: ParsonsCode,
        onsetTime: TimeInterval,
        duration: TimeInterval
    )
}

public enum ParsonsCode: String, Equatable, Hashable, Sendable, CaseIterable {
    case up      // *
    case down    // d
    case repeat_ // r  (named with trailing underscore to avoid Swift keyword)
}
```

Note: `repeat` is a Swift keyword. The trailing underscore is the established Swift convention. Display strings (`"*"`, `"d"`, `"r"`) match the standard Parsons code notation used in MIR literature for melodic similarity search.

### DetectedNote

The raw monophonic note event. Lower-level than ContourNote — these are what comes out of the pitch-tracking + onset pipeline before relative-pitch differencing produces the contour. MelodyKeyInference consumes these.

```swift
public struct DetectedNote: Equatable, Hashable, Sendable {
    /// MIDI note number (60 = middle C).
    public let midiNote: Int
    /// Pitch class (0–11; 0 = C).
    public var pitchClass: Int { midiNote % 12 }
    /// Onset time in seconds from the start of the analyzed buffer.
    public let onsetTime: TimeInterval
    /// Note duration in seconds.
    public let duration: TimeInterval
    /// Detection confidence (0.0–1.0) from the pitch tracker.
    public let confidence: Double

    public init(
        midiNote: Int,
        onsetTime: TimeInterval,
        duration: TimeInterval,
        confidence: Double
    )
}
```

The `pitchClass` is computed, not stored, because it's always `midiNote % 12` and never independently meaningful.

### MelodyKeyInference

Pitch-class-template-based key inference. Distinct from `ProgressionAnalyzer.inferKey` (which is chord-based). Consumes `[DetectedNote]`, returns ranked candidates.

```swift
public enum MelodyKeyInference {

    /// Infer the top key candidates from detected notes.
    /// Returns up to `maxCandidates` candidates ranked by score.
    /// Returns empty array if fewer than 3 notes or fewer than 2 distinct pitch classes.
    public static func infer(
        from notes: [DetectedNote],
        maxCandidates: Int = 2
    ) -> [KeyCandidate]

    public struct KeyCandidate: Equatable, Hashable, Sendable {
        public let key: MusicalKey
        /// Score 0.0–1.0: fraction of detected pitch classes that are diatonic to this key.
        public let score: Double
        /// Frequency count of how often the tonic appears in the input (used in disambiguation).
        public let tonicFrequency: Int

        public init(key: MusicalKey, score: Double, tonicFrequency: Int)
    }
}
```

Key differences from Cantus's current `MelodyKeyInference`:

- Uses MCC's `MusicalKey` directly rather than carrying around `String` representations.
- Returns `[KeyCandidate]` instead of a tuple of strings.
- `maxCandidates` is configurable; default 2 to preserve the "weak disambiguation = present two candidates" UX hint.
- Confidence thresholds (`kKeyConfidenceThreshold`, `kKeyLeadRatio`) move from constants on the type to configuration parameters. Recommendation: ship as internal constants in 0.0.8 (matching the `ProgressionAnalyzer` pattern from Q4 of 0.0.7), expose as configuration only when a consumer asks.

### OnsetDetector (DSP subsystem)

Lifts onset detection out of `AudioExtractor` and into the DSP layer where it belongs structurally. AudioExtractor uses it; future MCC capabilities (live onset tracking, AnalysisResult bundling) can also use it.

```swift
public enum OnsetDetector {

    /// Detect note onsets in a PCM buffer using RMS-energy thresholding against a running average.
    /// 
    /// Algorithm:
    /// - 2048-sample windows, 50% overlap (default; configurable)
    /// - Running average over last 10 frames
    /// - Onset declared when frame energy > running average × multiplier AND > absolute floor
    /// - Minimum gap enforced between successive onsets
    /// 
    /// - Returns: Onset times in seconds from buffer start.
    public static func detectOnsets(
        buffer: [Float],
        sampleRate: Double,
        configuration: Configuration = .default
    ) -> [TimeInterval]

    public struct Configuration: Equatable, Hashable, Sendable {
        public let windowSize: Int             // default 2048
        public let hopSize: Int                // default 1024
        public let minGapMs: Double            // default 500
        public let energyMultiplier: Float     // default 2.0
        public let energyFloor: Float          // default 0.005
        public let runningAverageWindow: Int   // default 10

        public init(
            windowSize: Int = 2048,
            hopSize: Int = 1024,
            minGapMs: Double = 500,
            energyMultiplier: Float = 2.0,
            energyFloor: Float = 0.005,
            runningAverageWindow: Int = 10
        )

        public static let `default`: Configuration
    }
}
```

This is energy-based onset detection, equivalent to Cantus's current algorithm. Spectral flux onset detection is documented as a future MCC enhancement (per the Capability Areas table) but explicitly out of scope for 0.0.8.

### NoiseCalibrator (DSP subsystem)

Lifts noise baseline calibration out of `AudioExtractor`. Same architectural argument as OnsetDetector.

```swift
public enum NoiseCalibrator {

    /// Scan a PCM buffer for genuine silence windows (RMS below threshold) and compute
    /// a noise baseline chroma vector. Returns nil if no genuine silence frames are
    /// present (continuous-playing recording).
    ///
    /// The baseline is suitable for subtraction from chord-detection chroma vectors
    /// to reduce noise contamination. A contamination safeguard is built in:
    /// if the resulting baseline has total chroma energy > 5.0, nil is returned
    /// (the calibration is rejected as contaminated).
    public static func calibrateBaseline(
        buffer: [Float],
        sampleRate: Double,
        windowSize: Int = 8192,
        hopSize: Int = 4096,
        silenceThreshold: Float = 0.001,
        minSilenceFrames: Int = 10,
        contaminationLimit: Double = 5.0
    ) -> NoiseBaseline?
}

public struct NoiseBaseline: Equatable, Hashable, Sendable {
    /// 12-element chroma vector representing average per-pitch-class energy in silence.
    public let chroma: [Double]
    /// Number of silence frames averaged to produce the baseline.
    public let frameCount: Int
    /// Total chroma energy (sum of all 12 elements). Used for contamination diagnostics.
    public var totalEnergy: Double { chroma.reduce(0, +) }

    public init(chroma: [Double], frameCount: Int)
}
```

The contamination safeguard threshold (5.0) and minimum silence frames (10) are exposed as parameters because they're empirical tuning values that may differ between Cantus's room-tone-friendly capture and Sanctuary's vocal capture.

- **Phase 2 stop point:** all algorithm tests pass; `MelodyKeyInference.infer` produces correct output for known fixture cases ported from Cantus's existing `MelodyKeyInferenceTests.swift`.

**Phase 3 — AudioExtractor pipeline (composition):**

- Composes Phase 1 primitives with existing `ChordDetector`, `ProgressionAnalyzer`, and `PitchDetector` from earlier releases. No new DSP work in this phase.
- **Phase 3 stop point:** end-to-end test on a known fixture buffer (e.g., a recorded C major scale or a I-IV-V-I progression) produces the expected `Result`.

**Cantus retirement (separate session, not part of 0.0.8 itself):**
- After 0.0.8 ships and Cantus adopts it, Cantus's local `AudioExtractor.swift` and `MelodyKeyInference.swift` are retired in favor of the MCC versions. Cantus keeps its `AudioEngine.swift` (audio session, recording state, UI bindings — explicitly consumer-side) and adds a thin adapter that hands the recorded buffer to `MCC.AudioExtractor.extract`.
- The D-major rescue rule (`dMajorRescue`) stays in Cantus. It's a guitar-voicing-specific heuristic for nylon-string strums and doesn't generalize to vocal or other instrumental capture. Sanctuary doesn't need it. Documenting it as Cantus-side logic prevents accidental promotion to MCC public API.

## Test coverage plan

**OnsetDetectorTests (~12 tests):**
- Single transient at known time produces single onset within window tolerance.
- Multiple transients spaced 1s apart produce all onsets in correct order.
- Two transients within the minimum gap produce only one onset (gap enforcement).
- Continuous tone with no transients produces no onsets.
- Silent buffer produces no onsets.
- Energy floor: a low-amplitude transient below the absolute floor is not detected.
- Configuration overrides: changing `minGapMs` from 500 to 100 allows previously-rejected onsets.
- Configuration overrides: changing `energyMultiplier` from 2.0 to 4.0 rejects previously-accepted weak onsets.
- Buffer too short (< window size) returns empty array.
- Sample-rate independence: same content at 44100 vs 48000 produces equivalent onset times within tolerance.
- First onset is allowed at sample 0 (no required initial gap).
- Public API regression: detector callable from outside the module via public `detectOnsets`.

**NoiseCalibratorTests (~10 tests):**
- Pure silence buffer produces a baseline with very low total energy (< 0.5).
- Buffer with only loud signal (no silence frames) returns nil.
- Buffer with mixed silence and signal: baseline computed from silence portion only; total energy below contamination limit.
- Contamination safeguard: explicit test where the silence-detection threshold is loose enough to admit signal frames; result is nil with `totalEnergy > contaminationLimit` checked.
- Minimum silence frames: buffer with 9 silence frames returns nil if `minSilenceFrames=10`.
- Custom silence threshold: a recording with only quiet ambient noise (above default threshold but below explicit higher threshold) produces a baseline.
- Sample-rate independence within tolerance.
- Public API regression: `calibrateBaseline` callable externally; `NoiseBaseline.totalEnergy` accessible.

**ContourNote / ParsonsCode / DetectedNote tests (~12 tests):**
- ContourNote public init produces accessible struct.
- ParsonsCode raw values match expected display strings ("*", "d", "r").
- DetectedNote.pitchClass computes correctly for all MIDI notes 0–127.
- Equality, hashing, Sendable for all three types.
- ParsonsCode.allCases returns expected three values in expected order.

**MelodyKeyInferenceTests (~20 tests, ported from Cantus):**
- Empty input returns empty.
- Single-note input returns empty.
- Two-note input with same pitch class returns empty (insufficient distinct).
- C major scale (C, D, E, F, G, A, B) → "C major" as top candidate, score 1.0.
- A minor scale (A, B, C, D, E, F, G) → "A minor" as top candidate, score 1.0.
- C major / A minor disambiguation by tonic frequency: notes with C appearing 4×, A appearing 1× → C major wins.
- C major / A minor disambiguation reversed: A appearing 4×, C appearing 1× → A minor wins.
- C major / A minor tied frequency: equal counts → A minor wins (minor preference tiebreak).
- maxCandidates=1 returns one candidate; =2 returns two (or fewer if ambiguous).
- Score reflects fraction-of-pitch-classes-diatonic correctly for partially-fit inputs.
- Public API regression for `MelodyKeyInference.infer` and `KeyCandidate`.

**AudioExtractorTests (~15 tests):**
- Single-segment fixture: known buffer with one chord, expected `Result.chordSegments.count == 1` with correct chord.
- Multi-segment fixture: I-IV-V-I in C major produces 4 segments with correct chord progression.
- Empty buffer returns empty result with duration=0.
- Buffer shorter than chroma window returns empty result.
- Configuration overrides: changing `extractionMinConfidence` from 0.25 to 0.50 rejects low-confidence segments.
- Configuration overrides: changing `earlyFrameWindowSize` changes segment chord output.
- Result.contour is empty when input is monophonic-with-no-pitch-tracking; populated when pitch detection succeeds.
- Result.detectedNotes correlates with Result.contour count.
- Result.key matches expected key for known fixtures.
- Pure silence buffer produces empty result (no onsets, no segments).
- End-to-end public API exercised from outside the module.

Estimated total: ~70 new tests. Existing test count after 0.0.7 was 143; 0.0.8 brings it to ~213.

## Open questions resolved

All six open questions resolved by Chris on 2026-04-25 per the recommendations in this spec.

### Q1. Configuration exposure — ship Configuration types now or defer?

**Resolved 2026-04-25:** Ship Configuration types now with Cantus-tuned defaults. Non-breaking for consumers to override at integration time. Sanctuary's vocal capture has different acoustic characteristics; exposing parameters now prevents Sanctuary from forking or tolerating Cantus-tuned defaults.

### Q2. Async or sync extract API?

**Resolved 2026-04-25:** Synchronous API only. Consumers wrap in `Task.detached` if they want background execution. The API is a pure function and synchronous is more testable; forcing async on a pure function locks consumers into an async call site for no benefit.

### Q3. ContourNote — how does the contour pipeline produce ContourNotes from DetectedNotes?

**Resolved 2026-04-25:** Option A — ship DetectedNote and ContourNote together in 0.0.8. AudioExtractor runs PitchDetector to produce frame-level pitches, segments into note events using onset boundaries plus stability windows, produces `[DetectedNote]`, then differences successive notes to produce `[ContourNote]`. Differencing is mechanical once DetectedNote exists.

### Q4. Onset detection — port energy-based as is, or upgrade to spectral flux?

**Resolved 2026-04-25:** Port energy-based as is. Spectral flux remains deferred per Capability Areas. Cantus's energy-based algorithm has been validated on hundreds of test recordings; spectral flux is a meaningful upgrade for noisy environments but complicates the OnsetDetector API. Right risk profile to ship as is and defer the upgrade.

### Q5. AnalysisPipeline subsystem placement — single module or sub-namespace?

**Resolved 2026-04-25:** Option A — top-level types under MusicCraftCore. The AnalysisPipeline folder is organizational only. Swift's namespacing through enum scoping is awkward; flat top-level types are friendlier. If future pipeline types create real namespace collisions, revisit then.

### Q6. Cantus adoption sequencing — how does Cantus retire its local AudioExtractor?

**Resolved 2026-04-25:** Cantus migration path documented in this spec. The actual Cantus-side migration is deferred to a separate session after 0.0.8 ships. Cantus keeps `AudioEngine.swift` and the recording lifecycle code; replaces the analysis call with `MCC.AudioExtractor.extract`. The D-major rescue is preserved by post-processing the MCC result inside Cantus. The 0.0.8 implementation prompt does not touch Cantus.

## Sanctuary-specific notes

Sanctuary's review on 2026-04-25 confirmed the API surface in full: ChordSegment fields sufficient (seconds-only timestamps, no chroma vectors needed), Result fields sufficient, ContourNote matches the frozen 2026-04-23 API, MelodyKeyInference.KeyCandidate sufficient, ParsonsCode.repeat_ naming confirmed. The slice-mapping in the original draft of this spec was incorrect; corrected mapping per Sanctuary's roadmap:

- **Slice 9** (MCC AudioExtractor integration): Sanctuary captures audio through AVFoundation, hands the buffer to AudioExtractor.extract, hydrates its AudioAnalysis schema (key, chord segments, contour) from the Result. MelodyKeyInference is implementation detail inside slice 9's hydration when the contour produces a useful pitch-class distribution.
- **Slice 10** (Fragment detail audio, with analysis): Consumes Result.chordSegments for the timeline highlight on the audio detail surface, plus Result.key and Roman-numeral annotation via MusicalKey.romanNumeralTyped(for:) (shipped 0.0.7). The Roman-numeral overlay on audio chord segments lives in slice 10, not slice 11.
- **Slice 11** (Fragment detail chord): ChordModality detail — typed/captured chord progression fragments with no audio buffer. Functionally unblocked by 0.0.7 ProgressionAnalyzer + romanNumeralTyped, NOT a 0.0.8 consumer. Sanctuary re-categorizes this off Phase C on its own roadmap.
- **Slice 12** (Takes / version history): Multi-take audio version history — canon designation, per-take auto-notes. Unrelated to MCC 0.0.8. Not a Phase C consumer.
- **Slice 15** (Hum-to-search): The genuine consumer of MelodyKeyInference outside slice 9. A brief hummed buffer's key needs disambiguation alongside DTW similarity match against stored fragment contours. KeyCandidate's default maxCandidates=2 works for this case.

Sanctuary-side schema absorption: AudioAnalysis.progression evolves from [Chord]? into the chord-segment timeline carrying per-segment timing and confidence; AudioAnalysis.contour concretizes from a stub to [ContourNote]. Tracked Sanctuary-side; pre-v1 yellow per Sanctuary's autonomy classification. Sanctuary will validate Configuration defaults against representative vocal recordings before slice 9 ships and override at the call site if needed.

## Out of scope / deferred

Explicitly NOT in 0.0.8 to keep scope controlled.

- **Audio file I/O.** AVAudioFile, format conversion, sample rate conversion. Consumer-side per the architectural boundary.
- **AVAudioEngine integration / live capture.** Consumer-side. MCC takes buffers.
- **Recording state / UI bindings.** Cantus keeps `@Published` recording properties in its own `AudioEngine.swift`. MCC's `AudioExtractor` is stateless.
- **D-major rescue rule.** Guitar-voicing-specific. Stays in Cantus, applied as post-processing on MCC's Result.
- **Spectral flux onset detection.** Deferred to a future MCC DSP enhancement.
- **Beat tracking, tempo estimation, downbeat detection.** Per the Capability Areas table — designed, deferred, candidate for 0.0.9 or 0.1.x.
- **Vocal feature extraction (range, tessitura, vibrato, voice type).** Per the Capability Areas table — designed, deferred.
- **Lyric extraction.** Per the Capability Areas table — designed, deferred to LyricsExtractor in 0.0.9 or 0.0.10.
- **AnalysisResult JSON-shaped bundled record.** Designed, deferred. The `AudioExtractor.Result` produced in 0.0.8 is the Swift-typed precursor; the JSON-shaped version comes later.
- **Configurable KeyInference / MelodyKeyInference weights.** Per the 0.0.7 Q4 decision pattern.
- **Cantus migration to MCC AudioExtractor.** Happens in a separate session after 0.0.8 ships.
