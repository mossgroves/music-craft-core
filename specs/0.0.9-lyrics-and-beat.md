# MCC 0.0.9 — LyricsExtractor and Beat Tracking Design Specification

Design document for the next MusicCraftCore release. Draft for review per the diagnosis-plan-execute pattern.

Status: Draft, awaiting Chris decision on iOS target and beat algorithm choice. Research phase complete; two independent subsystems scoped.

---

## Purpose

0.0.9 expands MCC's capabilities in two orthogonal directions: **voice/vocal analysis** (LyricsExtractor for on-device lyric transcription) and **rhythm analysis** (beat tracking and tempo estimation). Both are fully independent subsystems with no overlap in algorithms or APIs.

**LyricsExtractor** wraps Apple's Speech framework (SFSpeechRecognizer for iOS 17+ baseline, or SpeechAnalyzer for iOS 26+ capability) to produce timestamped word/phrase tokens that align to MCC's existing chord and melody timelines. This unlocks lyric-based search and analysis in consumer apps (Sanctuary's hum-to-search queries can now include lyric matching).

**Beat Tracking & Tempo Estimation** adds rhythm primitives to MCC's DSP layer: onset strength signal-based beat detection via autocorrelation, and tempo estimation from beat patterns. This unblocks Sanctuary's rhythm-aware music analysis and enables future structure segmentation (verse/chorus/bridge) in post-0.1.0 releases.

Both subsystems use existing infrastructure where possible (OnsetDetector from 0.0.8 can feed beat tracking; AudioExtractor's buffer interface works for both). No circular dependencies.

## Scope

In scope: a public `LyricsExtractor` enum in a new `Voice` subsystem wrapping Apple Speech framework; public `BeatTracker` and `TempoEstimator` in the DSP subsystem; new value types `TranscribedToken`, `Beat`, `TempoEstimate` following MCC conventions; comprehensive tests for both subsystems; research confirmation on Speech framework APIs.

Out of scope (deferred): voice activity detection (VAD — Apple's SpeechDetector handles this consumer-side), voice type classification (tenor/baritone/bass/soprano — future feature using vocal range + FHE thresholds), downbeat detection, meter/time signature inference, spectral flux-based beat detection upgrade.

## 1. LyricsExtractor API Surface

### Entry point

```swift
public enum LyricsExtractor {

    /// Transcribe speech from an audio buffer, producing timestamped word-level tokens.
    /// Async, wraps Apple's Speech framework (SFSpeechRecognizer or SpeechAnalyzer).
    /// Uses system-managed language models; no model bundling or management by MCC.
    ///
    /// - Parameters:
    ///   - buffer: Mono Float32 PCM samples
    ///   - sampleRate: Sample rate in Hz (typically 44100 or 48000)
    ///   - locale: BCP 47 language tag (e.g., "en-US", "es-ES"). Defaults to device locale.
    ///   - configuration: Optional tuning for speech detection (SpeechAnalyzer only; SFSpeechRecognizer has no config surface)
    /// - Returns: Array of timestamped tokens, or error if Speech framework is unavailable or transcription fails
    /// - Throws: SpeechFrameworkError (wrapped Apple errors)
    public static func transcribe(
        buffer: [Float],
        sampleRate: Double,
        locale: String = Locale.current.language.languageCode?.identifier ?? "en-US",
        configuration: Configuration? = nil
    ) async throws -> [TranscribedToken]

    /// Configuration for iOS 26+ SpeechAnalyzer only. Ignored when using iOS 17 SFSpeechRecognizer fallback.
    public struct Configuration: Equatable, Hashable, Sendable {
        /// Defer final results until the buffer ends, rather than returning partial hypotheses. Default: true.
        public let waitForFinalResult: Bool
        /// Include confidence scores for each token. Default: true.
        public let includeConfidence: Bool
        
        public init(waitForFinalResult: Bool = true, includeConfidence: Bool = true)
        public static let `default`: Configuration
    }

    public enum SpeechFrameworkError: Error, Equatable, Sendable {
        case frameworkUnavailable       // Speech framework not available on this device
        case recognitionFailed(String)  // Wrapped SFSpeechRecognitionError or SpeechAnalyzer error
        case localeUnsupported(String)  // Requested locale not supported
        case permissionDenied           // User denied microphone/speech recognition permission (consumer-side to handle)
    }
}

public struct TranscribedToken: Equatable, Hashable, Sendable {
    /// The recognized word or phrase.
    public let text: String
    /// Onset time in seconds from the start of the buffer.
    public let onsetTime: TimeInterval
    /// Duration of the token in seconds.
    public let duration: TimeInterval
    /// Confidence score 0.0–1.0. Present only if Configuration.includeConfidence = true.
    public let confidence: Double?
    /// Derived: offset time in seconds (onsetTime + duration).
    public var offsetTime: TimeInterval { onsetTime + duration }

    public init(
        text: String,
        onsetTime: TimeInterval,
        duration: TimeInterval,
        confidence: Double? = nil
    )
}
```

### Key design decisions on LyricsExtractor

1. **Async API.** Speech framework is inherently async. Forcing sync would require blocking on the async call, which defeats the purpose. Consumers wrap in Task if they want background execution (consistent with 0.0.8's AudioExtractor philosophy on async).

2. **Locale support.** Both SFSpeechRecognizer and SpeechAnalyzer support multiple languages. Defaulting to device locale ensures transcription works in Sanctuary's current locale context without requiring explicit setup. Fallback to "en-US" if locale detection fails.

3. **Separate Configuration for iOS 26+.** iOS 17's SFSpeechRecognizer has no configuration hooks at the API level (it's opaque). iOS 26's SpeechAnalyzer is more configurable. Rather than expose a config that only works on iOS 26, we accept it optionally and ignore it on older devices.

4. **Confidence score optional.** SFSpeechRecognizer does not expose per-token confidence (only overall recognition confidence on the transcription result). SpeechAnalyzer does. Making confidence optional accommodates both.

5. **Error handling.** All Speech framework errors (unavailable, permission denied, recognition failure) are wrapped in a SpeechFrameworkError enum. Permission handling is consumer-side (consumer's UIApplicationDelegate handles privacy requests).

6. **No model management.** Apple manages language models transparently. MCC doesn't bundle, load, or update Speech recognition models. Consumer apps may need to handle large model downloads (Apple's domain, not MCC's).

### Composition with AudioExtractor

Both LyricsExtractor and AudioExtractor take [Float] buffers and sampleRate. They are **fully independent**:
- LyricsExtractor feeds the buffer directly to the Speech framework (no pre-processing).
- AudioExtractor feeds the buffer to its own DSP pipeline (OnsetDetector, ChromaExtractor, etc.).
- A consumer app can call both on the same buffer: `let lyrics = try await LyricsExtractor.transcribe(...); let audio = AudioExtractor.extract(...);`
- No shared state, no data flow between them.
- Each has its own error path; one failing doesn't affect the other.

### iOS minimum target decision (OPEN QUESTION FOR CHRIS)

**Option A: iOS 17+**
- Use SFSpeechRecognizer (iOS 17+ baseline).
- Pros: Broader device compatibility, established API, stable.
- Cons: Older, less capable (no per-token confidence, less accurate on noisy audio).
- Estimated devices: ~95% of active iOS installs as of 2026.

**Option B: iOS 26+ (SpeechAnalyzer)**
- Use SpeechAnalyzer exclusively.
- Pros: Newer, more capable (per-token confidence, better accuracy, long-form support).
- Cons: Only iOS 26+ (cutting off devices from 2025 and earlier).
- Estimated devices: ~60–70% of active iOS installs post-iOS-26 adoption ramp.

**Recommendation for research purposes:** Option A (iOS 17+ with SFSpeechRecognizer) maximizes compatibility. If SpeechAnalyzer becomes critical for a consumer use case, Option B can be pursued in a 0.0.10 release targeting iOS 26+ only, or a compatibility layer can wrap both at runtime.

## 2. Beat Tracking & Tempo Estimation API Surface

### Entry points

```swift
public enum BeatTracker {

    /// Detect beat times in an audio buffer using onset strength signal autocorrelation.
    /// Stateless, pure function.
    ///
    /// - Parameters:
    ///   - buffer: Mono Float32 PCM samples
    ///   - sampleRate: Sample rate in Hz
    ///   - configuration: Optional tuning for onset detection and beat induction
    /// - Returns: Array of beat times in seconds, sorted by time. Empty if no beats detected.
    public static func detectBeats(
        buffer: [Float],
        sampleRate: Double,
        configuration: Configuration = .default
    ) -> [TimeInterval]

    public struct Configuration: Equatable, Hashable, Sendable {
        /// Onset detection window size (samples). Default: 2048.
        public let onsetWindowSize: Int
        /// Onset detection hop size (samples). Default: 1024 (50% overlap).
        public let onsetHopSize: Int
        /// Autocorrelation lag range (ms) to search for beat-period candidates. Default: 300–3000ms (~20–200 BPM).
        public let minBeatPeriodMs: Double     // default 300
        public let maxBeatPeriodMs: Double     // default 3000
        /// Minimum autocorrelation peak height (normalized 0–1) to consider a period a beat candidate. Default: 0.3.
        public let minAutocorrPeak: Double
        /// Dynamical system coupling constant for beat induction (0–1, higher = more inertia). Default: 0.5.
        public let inertia: Double

        public init(
            onsetWindowSize: Int = 2048,
            onsetHopSize: Int = 1024,
            minBeatPeriodMs: Double = 300,
            maxBeatPeriodMs: Double = 3000,
            minAutocorrPeak: Double = 0.3,
            inertia: Double = 0.5
        )
        public static let `default`: Configuration
    }
}

public enum TempoEstimator {

    /// Estimate tempo from beat times or from an audio buffer directly.
    /// If beats are provided, compute tempo from the inter-beat intervals.
    /// If buffer is provided, detect onsets and estimate tempo from the onset strength signal.
    ///
    /// - Parameters:
    ///   - beats: Pre-detected beat times in seconds. If provided, buffer is ignored.
    ///   - buffer: Audio buffer (used if beats is nil).
    ///   - sampleRate: Sample rate (required if buffer is provided).
    ///   - configuration: Optional tuning.
    /// - Returns: Array of tempo candidates ranked by confidence.
    public static func estimateTempo(
        beats: [TimeInterval]? = nil,
        buffer: [Float]? = nil,
        sampleRate: Double? = nil,
        configuration: Configuration = .default
    ) -> [TempoEstimate]

    public struct Configuration: Equatable, Hashable, Sendable {
        /// Same as BeatTracker.Configuration for onset detection. Used only if buffer is provided.
        public let onsetWindowSize: Int
        public let onsetHopSize: Int
        /// Autocorrelation lag range (ms) for tempo candidates. Default: 300–3000ms.
        public let minTempoMs: Double          // default 300
        public let maxTempoMs: Double          // default 3000
        /// Maximum number of tempo candidates to return. Default: 3.
        public let maxCandidates: Int
        /// Harmonic ratios to consider (e.g., [2, 0.5] includes half-tempo and double-tempo of the dominant). Default: [1, 2, 0.5, 1.5, 3, 0.33].
        public let harmonicRatios: [Double]

        public init(
            onsetWindowSize: Int = 2048,
            onsetHopSize: Int = 1024,
            minTempoMs: Double = 300,
            maxTempoMs: Double = 3000,
            maxCandidates: Int = 3,
            harmonicRatios: [Double] = [1, 2, 0.5, 1.5, 3, 0.33]
        )
        public static let `default`: Configuration
    }
}

public struct TempoEstimate: Equatable, Hashable, Sendable {
    /// Estimated tempo in beats per minute.
    public let bpm: Double
    /// Confidence score 0.0–1.0 (normalized autocorrelation peak height or inter-beat regularity score).
    public let confidence: Double
    /// Whether this tempo is a harmonic of a lower-frequency candidate (e.g., double-tempo). Allows consumers to group related tempos.
    public let isHarmonic: Bool

    public init(bpm: Double, confidence: Double, isHarmonic: Bool = false)
}
```

### Key design decisions on Beat Tracking & Tempo Estimation

1. **Autocorrelation-based algorithm.** Onset Strength Signal (OSS) autocorrelation is the established approach in academic literature (Ellis, Grosche, etc.) and is computationally tractable on-device. It captures both self-similarity (harmonic beat periods) and time regularity (regular inter-beat spacing). Spectral flux onset detection is deferred as a future DSP enhancement (per Capability Areas).

2. **Two entry points (beats vs. buffer).** BeatTracker takes a buffer and returns beat times. TempoEstimator can work from beats (faster, if beats are already detected) or directly from a buffer (one-shot). This composition pattern allows consumers to call BeatTracker once and TempoEstimator multiple times with different configurations, or skip BeatTracker entirely if they already have beats from another source.

3. **Harmonic ratio support.** Audio signals often have energy at tempos that are integer ratios of the true tempo (half-tempo due to rubato, double-tempo due to syncopation within a beat). TempoEstimate includes an `isHarmonic` flag and Configuration.harmonicRatios allows scoring related tempos. This helps consumers disambiguate when the signal is ambiguous.

4. **Inertia parameter in BeatTracker.** The inertia constant controls how much the beat induction algorithm resists changing tempo mid-stream. Higher inertia = more stable detected beats. This is a classical parameter in beat tracking literature (Viterbi tracking, dynamic programming). Exposed as configuration for tuning on different musical genres.

5. **Configuration defaults tuned for typical pop/rock.** BPM range 20–200 (300–3000ms beat period) covers typical music. Classical slow movements or electronic ambient music at <20 BPM will register as half- or quarter-tempo; this is a known limitation. If Sanctuary wants to support very slow music, Configuration.minBeatPeriodMs can be increased.

6. **Error handling.** No error path; returns empty arrays if no beats/tempos detected. This is appropriate because beat tracking is inherently uncertain (some audio is arrhythmic or too noisy to detect beats). Consumers check array length and decide whether to use the result.

### Composition with OnsetDetector

BeatTracker internally calls something equivalent to OnsetDetector to produce the onset strength signal, then applies autocorrelation. **Two options:**

**Option A (preferred for composition):** BeatTracker uses OnsetDetector.detectOnsets(...) internally, then autocorrelates the inter-onset intervals. Reuses the same onset detection tuning.
- Pro: Single onset detection code path, consistent tuning.
- Con: OnsetDetector returns onset times, not an onset strength signal; we'd need to convert times → inter-onset intervals → autocorrelation.

**Option B (simpler, less coupled):** BeatTracker implements its own onset strength computation internally (shorttime RMS-based, no dependency on OnsetDetector). Autocorrelates the strength signal directly.
- Pro: Self-contained, clear algorithm, no coupling to OnsetDetector configuration drift.
- Con: Duplicate onset detection logic.

**Recommendation for Chris:** Option B initially (self-contained, simpler), with a refactor in a future release if OnsetDetector evolution warrants sharing.

## 3. Subsystem Placement

**Voice subsystem (new):** Houses LyricsExtractor. Future voice/vocal analysis features (vocal range, tessitura, vibrato analysis, voice type classification) will expand this subsystem in later releases.

**DSP subsystem (expansion):** BeatTracker and TempoEstimator join the existing DSP primitives (PitchDetector, ChromaExtractor, OnsetDetector, etc.). No new subsystem; both beat tracking and tempo estimation are DSP-layer operations.

Architectural diagram (simplified):

```
┌──────────────────────────────────────────────────────────────┐
│ Voice (NEW in 0.0.9)                                         │
│   LyricsExtractor (wraps Apple Speech framework)             │
│     ├─ SFSpeechRecognizer (iOS 17+ path)                    │
│     └─ SpeechAnalyzer (iOS 26+ path, if chosen)             │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│ DSP (expansion in 0.0.9)                                     │
│   Existing: PitchDetector, ChromaExtractor, OnsetDetector... │
│   NEW: BeatTracker, TempoEstimator                           │
└──────────────────────────────────────────────────────────────┘

┌──────────────────────────────────────────────────────────────┐
│ AnalysisPipeline (0.0.8, unchanged)                         │
│   AudioExtractor (composes DSP + ChordDetection + Theory)    │
└──────────────────────────────────────────────────────────────┘
```

## 4. Test Coverage Plan

### LyricsExtractor Tests (~15 tests)

**Structural tests (no network, mocked Speech framework):**
- Empty buffer returns empty token array.
- Buffer with silence returns empty token array.
- Mock SFSpeechRecognizer result parsing: tokens with timestamps extracted correctly.
- Locale handling: default to device locale, override with explicit locale.
- Confidence score handling: present when available (iOS 26+), omitted on iOS 17.
- SpeechFrameworkError wrapping: permission denied, framework unavailable, recognition failure.

**Integration tests (iOS device / simulator only):**
- Real mic input through iOS 17 SFSpeechRecognizer (if using Option A).
- Real buffer transcription on iOS 26+ with SpeechAnalyzer (if using Option B).
- Multi-language support: English, Spanish, French (if SFSpeechRecognizer supports; SpeechAnalyzer definitely does).

**Note:** Testing Speech framework requires actual device or simulator with Speech framework support. Mock-only tests will cover the public API contract; real-audio accuracy tests deferred to a future fixture test suite (same pattern as AudioExtractor).

### BeatTracker Tests (~20 tests)

**Synthetic fixture tests:**
- Perfectly regular beat sequence (120 BPM constant) → detected beats match expected times within 50ms tolerance.
- Tempo variation (ritardando, accelerando) → beat induction follows the variation (tests inertia parameter).
- Polyrhythmic input (3+4 superposition) → detects the dominant beat period.
- Silence buffer → returns empty.
- Very slow tempo (40 BPM) → detected if minBeatPeriodMs set appropriately.
- Very fast tempo (180 BPM) → detected if maxBeatPeriodMs set appropriately.
- Configuration override: changing minAutocorrPeak affects detected beat count (lower threshold = more tentative beats).
- No beats in noisy random audio → returns empty or very few beats.
- Onset-strength signal edge case: first beat detection within first 1s of buffer.

**Note:** Real-audio beat accuracy is notoriously difficult to evaluate (human annotation is subjective). Deferred to a future real-audio fixture test alongside AudioExtractor tests.

### TempoEstimator Tests (~18 tests)

**From beats (faster path):**
- Regular beats 0.5s apart (120 BPM) → top candidate is 120 BPM.
- 15-beat sequence with constant spacing → 120 BPM confident.
- 5-beat sequence with low regularity → confidence score reflects the irregularity.
- Tempo variation: inter-beat intervals change over time → multiple candidates returned (one for slow/fast section).
- Harmonic ratios: input tempo 60 BPM with harmonicRatios=[1, 2, 0.5] → candidates at 60, 120, 30 BPM returned.

**From buffer (full path):**
- Call TempoEstimator.estimateTempo(buffer:...) with synthetic beat audio → top candidate near the expected tempo.
- Configuration override: changing harmonicRatios affects candidate count.
- maxCandidates=1 returns only top candidate; =5 returns up to 5.
- Empty buffer or silence → returns empty.

**Note:** Real-audio tempo accuracy requires calibrated test recordings (same caveat as beat tracking).

### PublicAPITests

- LyricsExtractor.transcribe is callable from external module.
- BeatTracker.detectBeats and TempoEstimator.estimateTempo are callable.
- TranscribedToken, TempoEstimate, Beat (if added), all public inits.
- All new public types conform to Equatable, Hashable, Sendable.

## 6. Open Questions for Chris

All of these require explicit decisions before implementation can proceed.

### Q1. iOS minimum target for LyricsExtractor (HIGH IMPACT)

**Decision needed:** iOS 17+ (SFSpeechRecognizer) or iOS 26+ (SpeechAnalyzer)?

| Aspect | iOS 17 | iOS 26 |
|--------|--------|--------|
| Market reach | ~95% of active installs | ~60–70% (ramp in progress) |
| Per-token confidence | No | Yes |
| Long-form audio support | No | Yes |
| API stability | Mature (8+ years) | New (WWDC 2025) |
| Model download mgmt | By Apple | By Apple (new improved) |

**Recommendation:** iOS 17 initially (broader reach), revisit in 0.0.10 if SpeechAnalyzer becomes critical. This gives iOS 26 ramp time and lets us validate Speech framework integration on production devices before upgrading.

### Q2. Beat tracking algorithm choice (MEDIUM IMPACT)

**Decision needed:** Autocorrelation-based (onset strength signal + autocorrelation) or alternatives?

Autocorrelation is established (Ellis, Grosche, Klapuri), implemented in Essentia and librosa, and tractable on-device (O(N log N) with FFT). Alternative approaches:
- **Tempogram + viterbi:** More sophisticated, tracks tempo changes, higher CPU cost.
- **BPM prior from music theory:** Genre/style-specific templates, requires training data or heuristics.
- **Machine learning (TCN, BLSTM):** State-of-the-art accuracy, requires model bundling (not portable to iOS without Core ML conversion).

**Recommendation:** Autocorrelation initially (proven, portable, low CPU). If Sanctuary wants higher accuracy on real audio, a post-0.1.0 release can add ML-based refinement via Core ML as an optional enhancement.

### Q3. Infrastructure sharing between BeatTracker and TempoEstimator (LOW IMPACT)

**Decision needed:** Do beat times automatically flow to tempo estimation, or are they fully independent?

**Option A (decoupled):**
- TempoEstimator.estimateTempo(buffer:...) ignores BeatTracker output.
- Each computes onsets internally.
- Simpler API, but redundant onset detection if both are called on the same buffer.

**Option B (composed):**
- TempoEstimator.estimateTempo(beats:...) takes detected beats as input (from BeatTracker or elsewhere).
- Reduces redundant computation if called in sequence.
- Requires consumers to call both in order, or accept the overhead of recomputing beats.

**Recommendation:** Option A for 0.0.9 (simpler, clearer ownership). If profiling shows redundant onset detection is a bottleneck, Option B can be added in 0.0.10 as a convenience overload without breaking existing APIs.

### Q4. Real-audio test fixtures for beat/tempo (LOWER PRIORITY)

**Decision needed:** Include real-audio fixtures in 0.0.9, or defer to 0.0.9.1 patch?

Real-audio ground truth is expensive to obtain (human annotation or synthetic generation with known MIDI). Deferring lets us ship faster and validate on production audio from Sanctuary/Cantus adoption.

**Recommendation:** Defer to 0.0.9.1 or 0.1.0. Synthetic fixture tests provide confidence that the algorithm works; real-audio tests provide confidence it works well. The former is a release blocker, the latter is a quality/performance concern.
